# TodoListApp 보안 정책 검토 보고서

본 문서는 프로젝트의 전체 보안 수준을 유지하기 위해 백엔드 구현 현황과 프론트엔드 설계 계획을 종합적으로 검토한 결과입니다.

---

## 1. 인증 및 인가 (Authentication & Authorization)

### 1.1 JWT 기반 인증 (공통)
- **방식**: Bearer Token 방식을 사용하여 무상태(Stateless) 인증을 구현함.
- **알고리즘**: `HS256`을 명시적으로 사용하며, `none` 알고리즘을 차단함.
- **토큰 관리 (FE 설계)**: JWT를 **Zustand 스토어 메모리**에만 저장함. `localStorage`나 `Cookie`를 사용하지 않아 브라우저 종료 시 세션이 만료되며, XSS를 통한 영구적 토큰 탈취 위험을 원천 차단함.

### 1.2 리소스 소유권 검증 (BE 구현)
- **원칙**: 모든 리소스(할일, 카테고리) 접근 시 `user_id`를 쿼리 조건에 포함하여 타인의 데이터 접근을 원천 차단함.
- **인가**: Service 레이어에서 리소스 존재 여부 확인 시 소유권을 함께 확인하며, 위반 시 404(Not Found)를 반환하여 리소스의 존재 유무 자체를 은닉함.

---

## 2. 데이터 보호 (Data Protection)

### 2.1 비밀번호 보안 (BE 구현)
- **해싱**: `bcrypt` 알고리즘과 `SALT_ROUNDS = 12`를 사용하여 강력한 일방향 해시를 적용함.
- **정책**: `validators.js`를 통해 최소 8자, 영문+숫자 혼합 정책을 강제함 (BR-10).

### 2.2 SQL Injection 방어 (BE 구현)
- **방식**: `node-postgres(pg)`의 파라미터화된 쿼리(`$1, $2, ...`)를 모든 데이터베이스 접근에 사용하여 SQL Injection 공격을 방어함.

### 2.3 입력 데이터 검증 (FE/BE 공통)
- **BE**: 모든 API 요청에 대해 형식 및 비즈니스 규칙 재검증 수행.
- **FE (계획)**: 폼 제출 전 클라이언트 사이드 유효성 검사를 통해 부적절한 데이터 전송을 1차적으로 차단하고 UX를 개선함.

---

## 3. 네트워크 및 세션 보안

### 3.1 CORS 및 DoS 방어 (BE 구현)
- **CORS**: 환경 변수(`CORS_ALLOWED_ORIGINS`)에 정의된 신뢰할 수 있는 도메인만 허용(White-list).
- **용량 제한**: 요청 본문 크기를 `10kb`로 제한하여 메모리 고갈 공격을 방어함.

### 3.2 세션 만료 및 자동 로그아웃 (FE 설계)
- **401 처리**: Axios Response 인터셉터를 통해 토큰 만료(401) 감지 시, 즉시 Zustand store를 초기화하고 로그인 페이지로 리다이렉트하는 자동 로그아웃 로직을 구현함.

---

## 4. 프론트엔드 특화 보안 계획 (FE Implementation Plan)

프론트엔드 개발 시 다음 보안 원칙을 준수하여 구현합니다.

1.  **XSS 방어**: React의 기본 데이터 바인딩(Auto-escaping) 기능을 활용하고, `dangerouslySetInnerHTML` 사용을 금지함.
2.  **환경 변수 관리**: `VITE_` 접두사를 사용한 환경 변수만 노출하고, 백엔드의 DB 비밀번호 등 민감한 비밀값이 프론트엔드 빌드 결과물에 포함되지 않도록 철저히 격리함.
3.  **에러 정보 은닉**: 서버에서 전달받은 `INTERNAL_SERVER_ERROR` 등의 상세 스택 트레이스를 사용자 UI에 직접 노출하지 않고 정형화된 메시지만 표시함.

---

## 5. 추가 권고 사항 (Roadmap)

1.  **Rate Limiting**: 로그인/회원가입 API에 대한 무차별 대입(Brute Force) 공격 방어 미들웨어 추가.
2.  **Secure Headers**: `helmet` 미들웨어를 BE에 추가하여 HSTS, CSP 등 브라우저 보안 헤더 강화.
3.  **HTTPS 적용**: 운영 환경 배포 시 전 구간 TLS 1.2 이상 적용 및 `Vite` 개발 서버의 HTTPS 옵션 검토.

---

**종합 검토 결과: 적합**
백엔드의 견고한 구현과 프론트엔드의 메모리 기반 토큰 관리 설계가 조화를 이루어, 개인 프로젝트 수준 이상의 높은 보안성을 확보할 것으로 판단됩니다.
