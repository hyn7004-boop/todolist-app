## UC-01 회원가입

### 시나리오 개요

처음 서비스를 접한 사용자가 이메일, 비밀번호, 이름을 입력하여 계정을 만들고, 기본 카테고리 3개(일반·업무·개인)가 자동으로 생성된다.

### 페르소나

- 이름: 박지훈
- 역할: 28세, 개발자 (신규 사용자)

### 사전 조건

- 사용자는 `/signup` 화면에 접근한 상태이다.
- 사용자는 아직 계정이 없다.

### 시나리오 흐름

1. 박지훈은 `/signup` 페이지에서 이메일 입력란에 `jihoon.park@example.com`을 입력한다.
2. 비밀번호 입력란에 `devLife9`를 입력하고, 이름 입력란에 `박지훈`을 입력한다.
3. "가입하기" 버튼을 클릭한다.
4. 시스템은 이메일 형식, 비밀번호 정책(8자 이상, 영문+숫자 각 1자 이상), 이름 길이(최대 50자)를 검증한다.
5. 시스템은 입력한 이메일이 이미 사용 중인지 확인한다.
6. 시스템은 계정을 생성하고, 동시에 기본 카테고리 3개(일반·업무·개인)를 자동으로 만든다.
7. 박지훈의 화면은 `/login` 화면으로 이동하며, "회원가입이 완료되었습니다. 로그인해 주세요." 안내 메시지가 표시된다.

### 사후 조건

- 박지훈의 계정(`status=active`)이 시스템에 생성되어 있다.
- 기본 카테고리 일반·업무·개인 3개가 해당 계정에 자동으로 생성되어 있다.
- 사용자는 로그인 화면으로 이동한 상태이다.

### 예외 시나리오

**예외 1: 이미 가입된 이메일 입력**

- 박지훈이 이미 다른 사람이 사용 중인 이메일 주소를 입력하고 "가입하기"를 클릭한다.
- 시스템은 이메일 입력란 아래에 "이미 사용 중인 이메일입니다."라는 안내 메시지를 표시하고, 가입을 진행하지 않는다.

**예외 2: 비밀번호 정책 미충족**

- 박지훈이 숫자 없이 영문자만으로 구성된 비밀번호(`devlife`)를 입력하고 "가입하기"를 클릭한다.
- 시스템은 비밀번호 입력란 아래에 "비밀번호는 8자 이상, 영문과 숫자를 각 1자 이상 포함해야 합니다."라는 안내 메시지를 표시한다.

## UC-02 로그인

### 시나리오 개요

기존 계정을 보유한 사용자가 이메일과 비밀번호를 입력하여 로그인하고, 할일 목록 화면으로 이동한다.

### 페르소나

- 이름: 이수현
- 역할: 35세, 마케팅 팀장

### 사전 조건

- 이수현은 이미 가입된 계정(`suhyun.lee@company.com`)을 보유하고 있다.
- 사용자는 `/login` 화면에 접근한 상태이다.

### 시나리오 흐름

1. 이수현은 출근 후 사무실 PC에서 브라우저를 열고 서비스 로그인 화면(`/login`)에 접속한다.
2. 이메일 입력란에 `suhyun.lee@company.com`, 비밀번호 입력란에 자신의 비밀번호를 입력한다.
3. "로그인" 버튼을 클릭한다.
4. 시스템은 이메일과 비밀번호의 일치 여부를 확인하고, 계정 상태가 활성(`active`)인지 검증한다.
5. 인증에 성공하면 서버는 JWT를 발급하고, 클라이언트는 이를 Zustand 메모리에 저장한다.
6. 화면이 할일 목록 화면(`/todos`)으로 전환되고, 상단에 "이수현님, 환영합니다."와 같은 인사 문구가 표시된다.

### 사후 조건

- 이수현은 인증된 상태로 할일 목록 화면에 있다.
- JWT가 Zustand 메모리에 유지되며, 브라우저를 닫거나 새로고침하면 초기화된다.

### 예외 시나리오

**예외 1: 비밀번호 불일치**

- 이수현이 비밀번호를 잘못 입력하고 "로그인" 버튼을 클릭한다.
- 시스템은 "이메일 또는 비밀번호가 올바르지 않습니다."라는 안내 메시지를 화면에 표시하고, 다시 입력할 수 있도록 폼을 유지한다.

**예외 2: 탈퇴한 계정으로 로그인 시도**

- 이수현이 이전에 탈퇴 처리된 계정의 이메일과 비밀번호를 입력하고 "로그인"을 클릭한다.
- 시스템은 "이메일 또는 비밀번호가 올바르지 않습니다."라는 동일한 안내 메시지를 표시한다. (탈퇴 계정 여부를 별도로 노출하지 않아 보안을 유지한다.)

## UC-03 로그아웃

### 시나리오 개요

로그인 중인 사용자가 현재 세션을 종료하고 로그인 화면으로 이동한다.

### 페르소나

- 이름: 이수현
- 역할: 35세, 마케팅 팀장

### 사전 조건

- 이수현은 현재 로그인된 상태로 할일 목록 화면(`/todos`)에 있다.

### 시나리오 흐름

1. 이수현은 퇴근 전 업무 마무리 후, 화면 상단 우측의 계정 메뉴를 클릭한다.
2. 드롭다운 메뉴에서 "로그아웃" 항목을 선택한다.
3. 클라이언트가 Zustand 메모리의 JWT를 초기화하고 TanStack Query 캐시를 제거한다.
4. 화면이 로그인 화면(`/login`)으로 전환되고, "로그아웃되었습니다."라는 안내 메시지가 잠깐 표시된다.

### 사후 조건

- Zustand 메모리의 JWT가 초기화되어 이수현의 인증 상태가 종료되어 있다.
- 브라우저는 로그인 화면에 위치한다.
- 이전에 접근하던 인증 필요 화면에 다시 접속하려 하면 로그인 화면으로 이동된다.

### 예외 시나리오

**예외 1: 이미 만료된 세션에서 로그아웃 시도**

- 오랜 시간 자리를 비운 이수현이 돌아와 로그아웃 버튼을 클릭하였으나 이미 세션이 만료된 상태이다.
- 시스템은 만료된 세션임을 감지하고, 별도의 오류 메시지 없이 로그인 화면으로 바로 이동시킨다.

## UC-04 개인정보 수정

### 시나리오 개요

로그인한 사용자가 내 정보 수정 화면에서 이름 또는 비밀번호를 변경한다.

### 페르소나

- 이름: 김영숙
- 역할: 43세, 부서장

### 사전 조건

- 김영숙은 로그인된 상태로 서비스를 이용 중이다.
- 화면 상단 메뉴 또는 설정 아이콘을 통해 `/settings` 화면에 접근한 상태이다.

### 시나리오 흐름

1. 김영숙은 `/settings` 화면에서 이름 변경 섹션을 찾는다.
2. 이름 입력란의 기존 값("김영숙")을 지우고 "김영숙(부서장)"으로 수정한다.
3. "저장" 버튼을 클릭한다.
4. 시스템은 이름 길이(최대 50자)를 검증한 후 변경 사항을 저장한다.
5. 화면 상단의 사용자 이름 표시가 "김영숙(부서장)"으로 즉시 반영되고, "정보가 업데이트되었습니다."라는 안내 메시지가 표시된다.
6. 이어서 비밀번호도 변경하기 위해 비밀번호 변경 섹션으로 이동한다.
7. 현재 비밀번호 입력란에 기존 비밀번호를 입력하고, 새 비밀번호 입력란에 새로운 비밀번호를 입력한다.
8. "비밀번호 변경" 버튼을 클릭한다.
9. 시스템은 현재 비밀번호 일치 여부를 확인하고, 새 비밀번호의 정책(8자 이상, 영문+숫자 각 1자 이상)을 검증하여 저장한다.
10. "비밀번호가 변경되었습니다."라는 안내 메시지가 표시된다.

### 사후 조건

- 김영숙의 이름이 "김영숙(부서장)"으로 저장되어 있다.
- 비밀번호가 새로 입력한 값으로 변경되어 있다.
- 다음 로그인 시 새 비밀번호로 로그인해야 한다.

### 예외 시나리오

**예외 1: 현재 비밀번호 불일치 (HTTP 400 수정됨)**

- 김영숙이 현재 비밀번호를 잘못 입력하고 "비밀번호 변경" 버튼을 클릭한다.
- 시스템은 현재 비밀번호 입력란 아래에 "현재 비밀번호가 올바르지 않습니다."라는 안내 메시지를 표시하고(HTTP 400 반환), 변경을 진행하지 않는다.
- 참고: BUG-001 수정으로 인해 `WRONG_CURRENT_PASSWORD` 에러 시 HTTP 401 대신 HTTP 400이 반환되며, Axios 인터셉터의 강제 로그아웃이 발생하지 않는다.

**예외 2: 새 비밀번호 정책 위반**

- 김영숙이 새 비밀번호로 숫자가 포함되지 않은 짧은 문자열을 입력한다.
- 시스템은 새 비밀번호 입력란 아래에 "비밀번호는 8자 이상, 영문과 숫자를 각 1자 이상 포함해야 합니다."라는 안내 메시지를 표시한다.

## UC-05 회원 탈퇴

### 시나리오 개요

더 이상 서비스를 이용하지 않으려는 사용자가 회원 탈퇴를 요청하고, 계정이 비활성화되며 자동으로 로그아웃된다.

### 페르소나

- 이름: 박지훈
- 역할: 28세, 개발자

### 사전 조건

- 박지훈은 로그인된 상태이다.
- `/settings` 화면에 접근한 상태이다.

### 시나리오 흐름

1. 박지훈은 `/settings` 화면 하단의 "회원 탈퇴" 섹션을 찾는다.
2. "회원 탈퇴" 버튼을 클릭한다.
3. 화면에 탈퇴 확인 다이얼로그가 표시된다. 다이얼로그에는 "탈퇴 후에는 동일한 이메일로 재가입이 불가합니다. 정말 탈퇴하시겠습니까?"와 같은 안내 문구와 함께 "탈퇴 확인" 및 "취소" 버튼이 있다.
4. 박지훈은 내용을 확인하고 "탈퇴 확인" 버튼을 클릭한다.
5. 시스템은 박지훈의 계정을 탈퇴 처리(비활성화)하고 현재 로그인 세션을 즉시 무효화한다.
6. 화면이 로그인 화면(`/login`)으로 이동되고, "회원 탈퇴가 완료되었습니다."라는 안내 메시지가 표시된다.

### 사후 조건

- 박지훈의 계정은 비활성화 상태로 전환되어 더 이상 로그인이 불가하다.
- 탈퇴에 사용한 이메일 주소로는 신규 가입이 불가하다.
- 기존에 등록했던 할일 및 카테고리 데이터는 서비스 내에서 접근할 수 없게 된다.

### 예외 시나리오

**예외 1: 탈퇴 확인 다이얼로그에서 취소**

- 박지훈이 탈퇴 확인 다이얼로그에서 "취소" 버튼을 클릭한다.
- 다이얼로그가 닫히고 `/settings` 화면이 그대로 유지된다. 계정에는 아무 변화가 없다.

## UC-06 카테고리 추가

### 시나리오 개요

사용자가 기존 기본 카테고리 외에 자신만의 새로운 카테고리를 추가하여 할일 분류 체계를 확장한다.

### 페르소나

- 이름: 박지훈
- 역할: 28세, 개발자

### 사전 조건

- 박지훈은 로그인된 상태이다.
- `/categories` 화면에 접근한 상태이다.
- 현재 카테고리 목록에 기본 카테고리 3개(일반·업무·개인)가 있다.

### 시나리오 흐름

1. 박지훈은 `/categories` 화면에서 현재 카테고리 목록(일반·업무·개인)을 확인한다.
2. 사이드 프로젝트 관련 할일을 따로 분류하고 싶어 "카테고리 추가" 입력란에 `사이드 프로젝트`를 입력한다.
3. "추가" 버튼을 클릭한다.
4. 시스템은 카테고리 이름이 50자 이내인지, 동일 계정 내에 같은 이름의 카테고리가 없는지 확인한다.
5. 새 카테고리가 생성되고, 카테고리 목록에 "사이드 프로젝트"가 추가되어 표시된다.

### 사후 조건

- 박지훈의 카테고리 목록에 "사이드 프로젝트" 카테고리가 추가되어 있다.
- 할일 등록 및 필터 화면에서 해당 카테고리를 선택할 수 있다.

### 예외 시나리오

**예외 1: 이미 존재하는 카테고리 이름 입력**

### 페르소나

**예외 1: 필터 조건에 맞는 할일이 없는 경우**

- 김영숙이 필터를 적용했으나 해당 조건에 부합하는 할일이 존재하지 않는다.
- 시스템은 오류 없이 빈 목록과 함께 "조건에 맞는 할일이 없습니다."라는 안내 문구를 표시한다.

**예외 2: 기간 필터 시작일이 종료일보다 늦은 날짜로 입력**

- 김영숙이 실수로 시작일(`2026-05-17`)을 종료일(`2026-05-13`)보다 늦게 입력하고 필터를 적용한다.
- 시스템은 기간 입력란 근처에 "시작일은 종료일보다 이전이어야 합니다."라는 안내 메시지를 표시하고, 조회를 진행하지 않는다.
